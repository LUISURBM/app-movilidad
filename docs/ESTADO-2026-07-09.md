## 3. Opciones de mejora

- **Verificación de fronteras de módulos en CI** (`dependency-cruiser` o `eslint-plugin-boundaries`): prohibir por build que un módulo alcance el dominio/tablas de otro. Barato ahora que son 7 contextos; endurece ADR-0001. *(Ya recomendado en ADR-0010.)*
- **Reglas de cumplimiento como datos/configuración** (inspiración DMN): umbrales del semáforo (30/15/3) y tipos de documento requeridos como config editable, no como código, para cambiarlos sin desplegar. Prepara un motor de reglas futuro sin acoplarse.
- **Medir cobertura de tests** (`@vitest/coverage`): hoy hay ~142 tests pero cobertura no medida → puntos ciegos desconocidos.
- **Observabilidad ligera:** logs estructurados, health checks, y rastreo de errores self-hosted (p. ej. GlitchTip/Sentry OSS) — E13 sigue delgado.
- **Capitalizar la investigación Spring/Java:** commitear una versión destilada del estudio en `docs/` (lo pide ADR-0010) para no perder el trabajo.
- **UX:** el portal tiene features "delgadas" (un `index.tsx` por área). A medida que entren pilotos, invertir en estados vacíos, mensajes de error accionables y onboarding guiado (clave para conversión en H1).

## 4. Riesgos y vulnerabilidades

### Técnicos / de proceso

| Riesgo | Impacto | Mitigación |
|---|---|---|
| **Deriva de documentación** (README dice "sin código de producción"; specs desactualizadas) | Confunde a stakeholders, agentes IA y nuevos devs; erosiona confianza | Truth-up (paso 2); bumpear estado de spec como parte del "Definition of Done" de cada PR |
| **Brecha UI móvil vs specs "Implemented"** | Sobreestimar la preparación para dogfooding | Cerrar UI de tanqueo/novedad/foto (paso 3) |
| **Cobertura de tests no medida** | Puntos ciegos antes de operar con datos reales | Activar cobertura (paso 6) |
| **Bus factor 1–3 personas** | Continuidad | Documentar runbooks; apoyarse en agentes; mantener ADRs |
| **Backups/restore no evidenciados** | Pérdida de datos en dogfooding | Backup + prueba de restauración (paso 4) |

### Seguridad (no son vulnerabilidades activas explotables hoy, sí puntos a vigilar)

- **Fallback de auth por headers** (`dev-auth.middleware.ts`): si se despliega sin `FLEETSPECIAL_JWT_SECRET`, la API confía en `x-tenant-id`/`x-roles` del cliente → suplantación de tenant/rol. El `docker-compose` obliga el secreto, pero un despliegue manual fuera del compose queda expuesto. **Recomendación: fallar cerrado siempre.**
- **SQLite móvil sin cifrar:** riesgo Habeas Data (Ley 1581/2012) al manejar datos de terceros. Pendiente SQLCipher.
- **APK firmado con debug key + `usesCleartextTraffic`:** aceptable solo para demo; nunca para producción.
- **Sin escaneo de dependencias ni lockfile congelado en CI:** deriva de cadena de suministro.

### Negocio / legal

- **Supuestos legales sin confirmar** (Habeas Data, facturación DIAN, Decreto 1079/2015 de transporte especial) marcados *"confirmar con abogado"* en `docs/01`. Riesgo regulatorio latente; conviene validación legal antes de comercializar (H1). *No soy abogado: esto es información, no asesoría legal.*
- **Dependencia de la empresa afiliadora** (planilla/documentación) sin integración formal: el MVP debe aportar valor por gestión propia; la costura de integración queda para después.

## 6. Consideraciones

- **La decisión de ADR-0010 (mantener NestJS/TS para H0–H1) es sólida:** reescribir un MVP probado justo antes de validarlo violaría el time-to-market y el principio "útil para un vehículo antes que para mil". El dominio es Clean Architecture y portable, así que una futura extracción a JVM sería refactor acotado, no reescritura.
- **La decisión estratégica más importante abierta es el _trigger_ T1 de ADR-0010:** ¿el equipo que sostendrá el producto es predominantemente TypeScript o JVM/Spring? Conviene **confirmarlo pronto**, porque invierte la premisa de "un solo cerebro".
- **Diferir billing, GPS live, DIAN e IA es correcto** — no adelantarlos. La disciplina anti-sobreingeniería del blueprint se está respetando.
- **El cuello de botella hacia el dogfooding es móvil + endurecimiento**, no capacidades nuevas de backend.

## 7. Conclusiones

FleetSpecial está en **muy buena forma para un MVP en bootstrapping**: backend maduro, multi-tenant con RLS real, núcleo offline robusto y portal funcional, todo con trazabilidad Negocio→Dominio→Spec→Código y decisiones registradas en ADRs. La calidad de ingeniería es notablemente alta para un equipo de 1–3 personas.

El camino al **dogfooding con la Duster** —el hito que valida el negocio— no pasa por más backend, sino por tres frentes acotados: **(1)** completar la UI móvil (tanqueo, novedad, foto), **(2)** endurecimiento ligero (cifrado del SQLite, auth fail-closed, backups) y **(3)** poner la documentación al día para que refleje la realidad del código.

La decisión de stack (NestJS/TS) está bien fundamentada; el único gran interrogante estratégico es **confirmar el perfil de talento (T1)** cuanto antes. Con esos frentes resueltos, el proyecto está listo para responder su pregunta de negocio de H0: *¿esto le ahorra dolor real a un transportador?*
